# 演示文稿设计与制作工作流 SOP · v2

> 基于国家电投体系内实际汇报项目复盘整理
> 适用场景：央企工作汇报、研究成果汇报、专题分析汇报、上级单位审核备查材料

---

## 工作流的真实定位

```
┌──────────────────────┐         ┌──────────────────────┐
│  研究阶段（你做）       │   →     │  PPT 工作流（AI 装配） │
│  full-research     │         │                       │
│  quick-research       │         │  把成稿视觉化为 PPT     │
│  形成报告/大纲         │         │                       │
└──────────────────────┘         └──────────────────────┘
        上游产物                          下游环节
```

**核心判断**：PPT 工作流是**视觉装配流水线**，不是**内容生成器**。

- 你的核心价值在研究阶段已经发挥（形成判断和论点）
- AI 在这里的核心价值是**抽取 + 减法 + 装配**
- 工作流的工时应该是**30-60 分钟**，不是好几个小时

---

## 启动前必读

进入工作流之前，请先看 [task-input-checklist.md](task-input-checklist.md)。这份清单解决"你应该带什么进来"的问题。

简要重述：
- **必带**：完整成稿 + 使用约束（听众/场景/时长/风格）
- **可带**：取舍指引、必须保留原文的部分
- **不带**：你已经"嚼过的"内容、预先删减的章节、视觉指令、页数要求

---

## 核心判断：先想清楚三件事

**1. PPT 在解决什么组织门槛**

- 现场宣讲 → 视觉冲击优先，文字稀疏
- 存底备查 → 内容完整性优先，每页可独立复原讲话
- 上报审核 → 政治表述准确性优先，所有引文必须经得起逐字核对
- 印刷成册 → 印刷规范优先

**2. 谁讲给谁听**

- 讲者级别决定语气（院领导用站位语 / 部门负责人用业务语）
- 听众级别决定深度（中层讲战略 / 一线讲案例）

**3. 是否落到具体业务**

- 落到 → 可以下沉到工作部署、项目名称
- 不落到 → 保持战略高度

## 工作流八阶段

### 第一阶段 · 接收任务 + 接收成稿

**用 AskUserQuestion 一次问清四个约束**（不再是 6 个）：

| 问题 | 影响 |
|---|---|
| 用途（宣讲/存底/审核/印刷）？ | 决定文字密度与视觉策略 |
| 时长（XX 分钟）？ | 决定页数（约 0.5-1页/分钟） |
| 听众级别 + 场景？ | 决定深度、语气 |
| 风格（标准汇报/学术分享/对外发布）？ | 决定视觉系统 |

**为什么不再问"讲者已有思路/提纲？"**——因为成稿就是答案，AI 应该直接读。

**为什么不再问"是否落到具体业务"**——成稿里有就讲，没有就不讲，AI 应该判断。

**关键动作**：用户应该在接收任务的同时给出成稿路径（或拖入文件）。AI 接到任务后第一件事就是读成稿。

### 第二阶段 · AI 读懂成稿 + 权威文本补充

**AI 的工作**：
- 通读用户给的成稿
- 识别成稿里的三类内容：判断/论点、支撑数据/引用、过渡内容
- **如果成稿引用了党政文件**：用 WebSearch + WebFetch 抓取权威原文做校对
- 整理 `_extracted/权威原文汇编.md`（如果有权威引用）

**用户的工作**：
- 提供内部不公开的材料（领导讲话、内部文件）
- 提供官方 LOGO 图片（绝不能让 AI 自制）
- 标注引用边界（哪些是公开口径、哪些是内部口径）

**关键产物**：
- 一份"AI 读到的成稿结构理解"——让用户确认理解对了
- `_extracted/权威原文汇编.md`（如有权威引用）

### 第三阶段 · 结构化抽取 + 版式适配建议（替代原"大纲编制"）

**这一阶段是 v2 的最大变化**。

**AI 做的事**：
- 从成稿里抽取每节的核心论点（1-3 条/页）+ 支撑数据
- 给每节匹配版式建议（从标准模板的 10 种版式中选）
- 提出"取舍建议"：哪些进 PPT、哪些可略、哪些不进
- **保留用户的核心表述原文**——重要判断不要替换措辞

**呈现给用户的形态**（一份 .md 文档）：

```markdown
## 我对成稿的结构抽取（请审定）

### P1 封面
- 主标题：[从成稿标题来]
- 副标题：[可选]

### P2 章节扉页 · 第一部分 · "XX"
（对应成稿第 1 章）

### P3 [页面标题]
- **版式建议**：双栏图表+表格
- **核心论点**（来自成稿 1.1 节）：[用户原文]
- **关键数据**：[XX 数字、XX 引用]
- **建议略去**：[过渡段]

### P4 [页面标题]
- **版式建议**：数据指标墙（4 个大数字）
- ...

## 取舍建议
- 必讲：第 1、2、4 章 → 12-14 页
- 略讲：第 3 章 → 3-4 页（你定）
- 不进 PPT：第 5 章、附录
```

**用户的工作**：
- 审定结构（必讲/略讲/不讲）
- 审定版式建议（同意/换种版式）
- 必要时修订核心表述

**版本控制**：保留 `结构抽取_V1.md` → `V2.md`，便于回溯。

**关键区别**（与 v1 的"大纲编制"相比）：
- v1：AI 替你想大纲 → 你审定 → 多轮修订
- v2：AI 抽你成稿的结构 → 你审定取舍 → 一般 1-2 轮就定

### 第四阶段 · 技术选型

**三种方案对比**：

| 方案 | 视觉效果 | 修改便利 | 适合场景 |
|---|---|---|---|
| **python-pptx** | 一般 | 较难 | 数据驱动批量生成（月度报表） |
| **直接画 pptx** | 灵活但易乱 | 需软件 | 后续需 PowerPoint 协作修改 |
| **HTML+CSS → PDF** | 极佳 | 文本编辑即可 | 复杂版式、精细排版、定稿存档 |

**默认选择**：HTML+CSS → PDF（覆盖 90% 场景）

**例外**：
- 后续需要 PPT 文件让同事改 → 选 pptx
- 每月跑一次的报表 → 选 python-pptx

### 第五阶段 · HTML 装配（用标准模板）

**核心要求**：用 `template/`，不要发明新版式。

**装配流程**：
1. 复制 `template/template.html` 到项目目录
2. 按"结构抽取"的页面顺序复制 `<section>` 块
3. 每页按版式建议套用对应的主体组件
4. 关键论点用 `key-point` 虚线绿框
5. 关键数据用 `<span class="accent">XX</span>` 染红
6. 含数据的页加 `data-source` 数据来源

**v2 标准模板提供的版式**（详见 `template/sample-pages.html`）：
- 封面、章节分隔页
- 全宽图表、双栏图表+表格
- 编号点列、数据指标墙、三栏卡片
- 时间线、双栏文字结论
- 尾页

### 第六阶段 · 质量保证（双轨校验）

**校验 1 · AI 独立校验**

用子智能体（subagent）做独立校验：

```
任务：把PPT中每条原文引用与权威原文汇编逐字比对
输出：差异清单 + 严重程度（高/中/低）
```

**校验 2 · 人工审定**

AI 校验不能替代的：
- 政治表述合规性（特别是总书记称谓、党中央表述）
- 内部口径（哪些数字能讲、哪些不能讲）
- 保密边界

**关键校验点**：
- 党政文件引用是否完整准确
- 数字是否准确（百分比、亿元、装机量等）
- 关键定语是否缺失（"显著""基本""加快""全面"等）
- 节选必须加省略号
- 引用归属是否清晰

### 第七阶段 · 导出 PDF + 反向验证

**导出命令**（Chrome/Edge headless）：

```bash
chrome.exe --headless=new --disable-gpu \
  --no-pdf-header-footer \
  --print-to-pdf="C:\\absolute\\path\\output.pdf" \
  --no-sandbox \
  --virtual-time-budget=15000 \
  "file:///C:/absolute/path/source.html"
```

**反向验证**（必做）：

```python
import fitz
doc = fitz.open('output.pdf')
for idx in range(len(doc)):
    pix = doc[idx].get_pixmap(matrix=fitz.Matrix(1.2, 1.2))
    pix.save(f'_preview/p{idx+1:02d}.png')
```

把渲染的 PNG 看一遍——AI 视觉可以判断是否有重叠、溢出、字体显示问题。**这一步交付前主动做，不要等用户发现**。

### 第八阶段 · 归档与交付

**标准交付物**：

```
项目根目录/
├── 成品PPT.html              # HTML源文件（可后续修改）
├── 成品PPT.pdf               # 定稿PDF
├── 结构抽取_V2.md             # 抽取过程文档（思路备查）
├── _extracted/               # 权威原文汇编（如有）
└── _preview/                 # PDF抽样PNG（验证用）
```

## 四条铁律

### 铁律一：LOGO 绝不自制

党徽、企业 LOGO 必须从官方源/用户提供获取，存放在 `shared/logos/`。
- 党徽 → 共产党员网（12371.cn）
- 国家电投 → spic.com.cn
- 经研院 → 用户提供官方 VI 资源包

### 铁律二：党政文件引用必须从官方源抓取

不能凭记忆、不能用二手解读：
- 公报 → 新华社 / 中国政府网受权发布版
- 《建议》/《纲要》 → 新华社全文 + 人大网官方版

**典型坑**：党课PPT某页漏字"人民生活更加幸福美好"——就是抓取时省略导致。

### 铁律三：AI 做减法不做加法

进入工作流的内容应该是**用户研究产物的浓缩**，不是 AI 创造性添加：
- 用户的核心判断要原文保留
- AI 不要"补充新观点"
- 过渡内容（"综上""下面"）可以删
- 重要数据/引文必须保留

**违反代价**：AI 加的内容稀释用户的核心，反而拉低专业感。

### 铁律四：进入 PPT 工作流后不改研究内容

如果在装配过程中发现研究层面的问题（数据错了、判断不准），**退回研究阶段重做**，不要在 PPT 工作流里改内容。

**原因**：研究层面的修正需要重新做审慎判断，在 PPT 工作流里改容易引入新错误。

## AI 与人工的分工边界（v2 修订）

| 环节 | AI 主导 | 人工主导 |
|---|---|---|
| 任务定位 |  | 用途、听众、时长、风格 |
| 读懂成稿 | 通读、识别三类内容 | 提供成稿、标注引用边界 |
| 结构化抽取 | 抽核心、提建议、做减法 | 审定取舍、判断详略 |
| 版式适配 | 按内容性质推荐版式 | 同意或换种版式 |
| 视觉装配 | HTML/CSS 编码、套模板 |  |
| 质量校验 | 引文准确性、布局检查 | 政治合规、保密边界 |
| 导出归档 | PDF 导出、反向验证 |  |

**关键变化**（v1 → v2）：
- v1："大纲编制"是 AI 起草 + 用户审定（多轮迭代）
- v2："结构化抽取"是 AI 读用户成稿 + 推荐取舍（一般 1-2 轮）

## 时间分配参考（v2 修订）

**前提**：用户带着完整成稿进入工作流。

| 阶段 | 占比 | 实际时间 |
|---|---|---|
| 任务定位 + 4 个约束确认 | 5% | 2-3 分钟 |
| AI 读懂成稿 + 权威文本补充 | 15% | 5-10 分钟 |
| 结构化抽取 + 用户审定 | 20% | 8-12 分钟 |
| HTML 装配 | 35% | 15-20 分钟 |
| 质量校验 + 修订 | 15% | 5-10 分钟 |
| 导出 + 反向验证 + 归档 | 10% | 5 分钟 |

**总计**：约 40-60 分钟（一份 20-30 页的标准汇报）

**v1 → v2 节省时间的关键**：第三阶段从"多轮大纲迭代"变成"一次抽取 + 取舍审定"，节省 30-40 分钟。

## 五个反模式（v2 修订）

### 反模式 1：进了 PPT 工作流又改研究内容

发现研究有问题→退回研究阶段。在 PPT 工作流里改内容容易引入新错误。

### 反模式 2：让 AI 在你的研究基础上"扩充"内容

AI 加的内容稀释你的核心判断。让 AI 只做减法不做加法。

### 反模式 3：把研究报告"提前嚼一遍"再给 AI

你做了 AI 该做的工作，浪费时间。直接给完整成稿。

### 反模式 4：分多次零碎扔素材

AI 每次接收都要重新建立上下文。一次性把所有素材分级标注（主输入/参考/背景）一起给。

### 反模式 5：在 HTML 装配阶段反复调整版式

版式应该在第三阶段（结构化抽取 + 版式适配建议）就定下来。装配阶段开始反复换版式，工时翻倍。

## 写给下一个项目的自己（v2）

1. **接到任务先问 4 个约束，不要急着动手**（用途/时长/听众/风格）
2. **第一件事是读用户的成稿**——不要让用户重新提炼论点
3. **抽取后让用户审定取舍**——再装配 HTML
4. **权威文本必须从官方源完整抓取**——不能凭记忆写
5. **官方 LOGO 必须从用户/官方获取**——绝不自制
6. **AI 做减法不做加法**——保留用户的核心表述原文
7. **每页装配前确认版式**——装配阶段不再换版式
8. **导出 PDF 后主动用 PyMuPDF 反向验证**——不要等用户发现重叠
9. **进入 PPT 工作流后不改研究内容**——研究问题退回研究阶段
10. **总时长目标 40-60 分钟**——超过 90 分钟说明流程跑错了
